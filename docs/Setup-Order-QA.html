AWS & Terraform

How do I check my current IAM user on AWS CLI?

aws sts get-caller-identity

Shows Account, UserId, and Arn (user or role). List access keys (not the secret):

aws iam list-access-keys --user-name YOUR_IAM_USER_NAME

Note: AWS never shows an existing secret access key again. Create a new key if you lost it.

Is DynamoDB set up for Terraform state locking?

Not in the main stack by default. The main backend.tf uses S3 only unless you add dynamodb_table via backend.hcl.

Use terraform/bootstrap/ to create the state bucket and optional lock table (datavault-terraform-locks), then:

terraform output -raw backend_hcl > ../backend.hcl
cd ..
terraform init -backend-config=backend.hcl

After updating the S3 bucket name, is terraform apply all I need?

No. The state bucket must exist before terraform init. Full order:

  1. Bootstrap: terraform/bootstrap → apply → generate backend.hcl
  2. Main: terraform init -backend-config=backend.hcl
  3. terraform apply (with ssh_key_name in terraform.tfvars)
  4. Update manifests, GitHub, Argo CD (steps 5–8)

Can Terraform create the state bucket instead of the CLI?

Yes, via the separate terraform/bootstrap/ module (local state). The main stack cannot create its own remote backend on first run (chicken-and-egg). Bootstrap once, then point the main stack at that bucket.

How do I get or set ssh_key_name?

It is the name of an existing EC2 key pair in eu-west-2, not created by Terraform.

aws ec2 describe-key-pairs --region eu-west-2
aws ec2 create-key-pair --key-name datavault-key-pair --region eu-west-2 \
  --query KeyMaterial --output text | Out-File -Encoding ascii datavault-key-pair.pem

Set in terraform/terraform.tfvars:

ssh_key_name = "datavault-key-pair"

SSH: ssh -i datavault-key-pair.pem ec2-user@<EC2_PUBLIC_IP>

terraform apply failed on security group descriptions

AWS rejects Unicode em dashes (—) in security group rule descriptions. Use ASCII hyphens (-) instead, e.g. SSH - admin access to EC2.

GitHub & CI

Is a specific IAM user defined in the repo?

No. Terraform creates an IAM role for EC2 (datavault-k3s-ec2-role). GitHub Actions uses repository secrets you create (e.g. IAM user datavault-github-actions). Only secret names are in ci.yml.

I used the wrong Git remote (datavault-gitops). How do I fix it?

git remote set-url origin https://github.com/nokafor2/DataVault-Deployment-Automation.git
git remote -v
git push -u origin main

Update argocd/application.yaml repoURL to match, then commit and push.

What does git commit --allow-empty -m "Trigger first CI build" do?

Creates a commit with no file changes so you can push and trigger CI when you have nothing else to commit.

Caveat: Your workflow only runs when paths under app/, tests/, k8s/, or .github/workflows/ change. An empty commit may not trigger CI. Prefer pushing a real change (e.g. app/ or workflow file) or use workflow_dispatch.

CI is green but nothing commits / deploy manifest not updating

The bot commit happens in the build-and-deploy job after tests, ECR push, and manifest edit. Common blockers:

  • Tests failed → deploy job skipped
  • Empty / argocd-only commits → workflow not triggered (path filters)
  • Missing AWS secrets → ECR push fails
  • Dockerfile parse error → inline comments on FROM line (fixed by moving comments to separate lines)
  • httpx 0.28+ → pytest/TestClient failure (pin httpx<0.28 in CI)

Explain: check ECR, then git pull if CI updated the manifest

Green Actions means: tests passed, image pushed to ECR, and k8s/deployment.yaml updated on GitHub by github-actions[bot].

CheckWhy
ECR → datavault-apiConfirms image exists (pods need it)
git pull on laptopSyncs local repo with bot’s manifest commit

CI does not run kubectl. Argo CD or manual kubectl apply on EC2 deploys to the cluster.

Docker build failed: FROM requires either one or three arguments

Inline comments on the same line as FROM break the Docker parser. Put comments on their own lines:

# Minimal Python base
FROM python:3.11-slim

EC2 access

EC2 Instance Connect does not open / SSH connection failed

  • Pop-up blocker — allow pop-ups for AWS Console
  • Username — use ec2-user on Amazon Linux 2023
  • Prefer SSH with .pem — ssh -i datavault-key-pair.pem ec2-user@<PUBLIC_IP>
  • Session Manager — needs SSM agent online (may take several minutes after boot)

Instance Connect can fail even when port 22 is open if ephemeral keys are not accepted; user_data.sh was updated to install ec2-instance-connect and restart sshd on new instances.

Setup order (steps 5–8)

Where do I replace 000000000000 in deployment.yaml?

Line 32 — the image: field:

image: 791778419317.dkr.ecr.eu-west-2.amazonaws.com/datavault-api:latest

Get account ID: terraform output aws_account_id

Should I run kubectl apply -f k8s/ on my EC2 server?

Yes — kubectl must target the k3s API on the EC2 node (after export KUBECONFIG=/etc/rancher/k3s/k3s.yaml).

Clone the repo on EC2, then either:

  • Path A (GitOps): kubectl apply -f argocd/application.yaml — Argo CD syncs k8s/ from Git
  • Path B (manual): kubectl apply -f k8s/ first, then register Argo CD

Step 7 (image in ECR) should complete before step 8.

kubectl & cluster commands

What does export KUBECONFIG=/etc/rancher/k3s/k3s.yaml do?

Tells kubectl which cluster to use (the k3s config written at install time). Run on EC2 after SSH. Alternative: sudo k3s kubectl get nodes.

What does kubectl apply -f k8s/ do?

Creates or updates all manifests in k8s/: Deployment, Service, ConfigMap, Secret, HPA. Use for manual first deploy; with Argo CD, Git changes usually replace this.

What does kubectl get pods -w do?

Lists pods and watches live. Look for Running and 1/1 READY. Ctrl+C to stop. Use kubectl logs <pod> if status is ImagePullBackOff or CrashLoopBackOff.

What do the Argo CD + health check commands do?

kubectl apply -f argocd/application.yaml
kubectl get pods -w
curl -s http://localhost:30080/health
  • apply argocd/ — register Git repo/path so Argo CD syncs k8s/ (run once)
  • get pods -w — watch until app pods are healthy
  • curl :30080/health — hit API via NodePort on the EC2 host

Argo CD — apply and verify

How do I execute and check “apply argocd/application.yaml so Git owns future deploys”?

Execute (once on EC2):

export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
git clone https://github.com/nokafor2/DataVault-Deployment-Automation.git
cd DataVault-Deployment-Automation
kubectl apply -f argocd/application.yaml

Check:

kubectl get application -n argocd
# Want: SYNC STATUS = Synced, HEALTH STATUS = Healthy

kubectl get pods
curl -s http://localhost:30080/health

UI: http://<EC2_PUBLIC_IP>:30081 — user admin, password from:

kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d; echo

After this: pushes to main → CI updates k8s/deployment.yaml → Argo CD syncs → no SSH needed for routine deploys.
